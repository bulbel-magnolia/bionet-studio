# bionet-studio

> 一个通用的合成生物网络工作台：搭建线路网络，配置输入、节点和边，选择读出与决策规则，进行可复现的前端仿真。

bionet-studio 的核心想法是把生物线路抽象成可编辑的网络：输入层接收环境或实验信号，调控层和响应层整合信号，输出层给出一个或多个读出。平台不绑定某一个应用场景；信号整合、逻辑门、多输出报告器、甘草质量检测等都只是可替换的示例项目。

所有默认数值都按 **illustrative / 待标定** 处理。平台当前用于设计推演、界面验证和模型工作流展示，不宣称给出湿实验级定量预测。

## 怎么本地运行

首次拉取后安装依赖：

```powershell
npm install
```

开发预览：

```powershell
npm run dev
```

生产构建：

```powershell
npm run build
```

提交前完整检查：

```powershell
npm run verify
```

该命令会依次运行类型检查、基础测试、生产构建和 `docs/` 静态发布目录自检。构建后 `docs/` 是纯静态 HTML / CSS / JS，可继续作为 GitHub Pages 或 iGEM 静态托管目录。

GitHub Actions 模板见 `ci/github-actions-ci.example.yml`。启用时需要用具备 `workflow` 权限的 GitHub 账号复制到 `.github/workflows/ci.yml`。

## 核心模型

| 平台对象 | 生物含义 | 当前实现 |
|---|---|---|
| Sensor / Input | 输入信号或感受元件 | Hill 响应，将输入水平转成活性 |
| Regulator | 中间调控或信号中继 | 加权输入 + 非线性响应 |
| Response node | 响应层隐藏节点 | 整合多路信号 |
| Reporter / Output | 输出通道 | 带表达与成熟动力学的读出 |
| Edge weight | 调控强度或串扰关系 | 正/负权重，激活或抑制 |
| Readout | 用户关心的输出解释方式 | 曲线、分类、热图、剂量响应、真值表 |

当前仿真流程概括为：

```text
输入水平
  → Hill 编码
  → 节点间加权传递
  → 非线性响应
  → 一阶动力学
  → 输出通道 / 决策规则 / 读出面板
```

完整规划见 [`平台规划/01_总体规划_v0.1.md`](平台规划/01_总体规划_v0.1.md)。

## 第一轮任务布置

更新时间：2026-06-11

DDL：2026-06-18

本轮工作的目标，是在现有高保真前端原型的基础上，完成一次工程收口，使 bionet-studio 从“可展示的原型”进一步推进为“结构清晰、结果可复现、后续可协作扩展”的软件项目。本轮暂不追求一次性实现全部路线图功能，而是优先稳定项目底座。

本轮工作的三个重点如下：

1. 工程可复现：项目结构、仿真逻辑、导入导出流程需要保持稳定。
2. 交互可演示：用户应能顺利完成搭建网络、修改参数、查看结果的基本流程。
3. 生物数据可信：示意值、待标定值和真实来源参数必须清楚区分。

**一、前端交互负责人：张森威**

张森威负责前端交互与工具体验，重点是让当前工作台从“可以观看”进一步变成“可以操作、可以演示、可以被陌生用户理解”的软件工具。

本轮主要任务包括：

1. 检查并打磨画布交互。  
   需要检查节点拖动、节点选择、边选择、权重修改、激活/抑制切换、节点复制与删除、边删除等基础流程。重点关注用户能否清楚判断当前选中的对象，以及能否顺利完成基本网络编辑操作。

2. 设计或实现 A/B run 对比的第一版。  
   目标是让用户能够保存一次当前运行结果，在修改输入或参数后，与新的运行结果进行对比。第一版可以先实现曲线叠加和关键指标对比，例如 final output、peak、state change 和 delta。该功能优先服务演示场景，用于体现平台对参数调整和设计决策的帮助。

3. 整理读出面板体验。  
   需要检查 time series、output channels、decision rule、layer activity、dose-response、truth table 等读出模块是否容易理解，添加和配置入口是否明显，中文界面是否存在拥挤或溢出问题。

本轮交付物：

1. 画布交互问题清单与改进建议。
2. A/B run 对比的交互方案，或一个可演示的第一版实现。
3. 读出面板体验检查结果。

验收标准：

1. 新用户能够在较短时间内完成添加节点、连接节点、选择边、修改权重等基本操作。
2. 能够演示“修改参数前后，输出结果发生变化”的对比流程。
3. 中文界面不出现明显溢出、遮挡或难以理解的按钮。

**二、工程内核负责人：郎咸欣**

郎咸欣负责工程内核与项目结构，重点是保证模型计算、数据结构、导入导出和基础测试的可靠性。该方向决定了后续功能是否能够稳定扩展。

本轮主要任务包括：

1. 整理 project schema 草案。  
   需要明确一个项目包含哪些字段，例如 meta、nodes、edges、params、aggregate、readouts。不同类型节点需要哪些参数，每条边需要哪些字段，每个 readout 如何配置，都应形成清晰说明。该 schema 将作为前端交互、元件库接入和后续反向设计的共同接口。

2. 给 sim.js 补基础测试。  
   测试至少应覆盖 Hill 响应、激活边、抑制边、classification 阈值、memory latch 等关键行为。测试不需要复杂，但应能快速发现核心仿真逻辑是否被改坏。

3. 检查 JSON 导入导出流程。  
   需要确认导出的 JSON 再导入后可以完整恢复项目，包括节点、边、参数、readouts 和 aggregate。对于缺少部分字段的旧版本 JSON，应有基本 fallback，避免页面直接失效。

4. 保持仿真内核与界面解耦。  
   `sim.js` 应以 project 作为输入，输出 simulation result，不直接依赖 DOM 或界面状态。这样后续反向设计、批量运行和测试都可以复用同一个仿真内核。

本轮交付物：

1. project schema 草案。
2. sim 基础测试脚本或测试说明。
3. JSON 导入导出检查结果。

验收标准：

1. 前端能够根据 schema 明确读取和修改哪些字段。
2. 修改 `sim.js` 后，有基础测试可以验证关键行为。
3. 一个导出的项目在刷新页面后再次导入，结果应保持一致。

**三、生物数据与文档负责人：汪奇**

汪奇负责生物数据、元件库框架与文档解释，重点是保证平台的生物学表述可信，避免将示意值误写为真实参数。该方向直接关系到平台在 iGEM 软件项目中的可信度和可复用性。

本轮主要任务包括：

1. 设计元件库字段表。  
   需要明确真实元件进入平台时应包含哪些信息，例如 element_id、name、type、biological_role、layer、parameters、source、confidence、calibration_status、notes。特别要明确参数来源与标定状态。

2. 整理第一批元件类别。  
   本轮优先整理类别，不急于填写大量真实参数。建议从 sensor、regulator、promoter / response node、reporter、memory / latch 几类开始，说明它们在平台中对应什么对象，需要哪些参数，以及哪些参数较难查证。

3. 给示例项目写解释。  
   先解释四个示例：Signal integration model、Logic AND gate、Multi-output reporter、Licorice quality example。说明它们分别展示什么能力，哪些数值是示意值或待标定值。特别需要强调，甘草质量检测是 example project，而不是平台本体的固定领域。

4. 起草数据纪律说明。  
   明确真实元件必须有来源，无来源参数只能标注为 illustrative / 待标定。平台输出是设计推演，不替代湿实验验证。

本轮交付物：

1. 元件库字段表。
2. 第一批元件类别说明。
3. 四个示例项目解释。
4. 数据纪律草案。

验收标准：

1. 每个参数都能回答“来源是什么”，或明确标注为示意值 / 待标定值。
2. 任一应用示例都不被写成平台本体；甘草质量检测也只是可替换示例。
3. 文档能够让其他团队理解平台的通用用途、示例含义和可信边界。

本轮工作虽然按负责人划分，但三条主线需要保持接口一致。

## 近期路线

短期重点是把当前高保真原型收束成更可靠的软件项目：

1. 稳定模型 schema：明确 project / node / edge / readout 的数据结构。
2. 补测试：给 `sim.js` 增加基础数值回归测试，防止后续 UI 改动影响仿真结果。
3. 元件库 schema：建立真实元件进入平台前必须携带的来源、置信度、待标定标记。
4. 生产工程化：在现有 Vite + React + TypeScript 工程壳上，逐步把 `src/legacy/` 拆成强类型模块。
5. 功能扩展：运行历史、A/B 对比、反向设计、真实元件推荐。

## 数据纪律

本项目对生物参数保持保守口径：

1. 没有来源的具体数值不能写成真实参数。
2. 默认示例值必须标注为 illustrative / 待标定。
3. 真实元件库后续必须记录 Registry、DOI、论文链接或实验来源。
4. 平台输出是设计推演，不替代湿实验验证。
