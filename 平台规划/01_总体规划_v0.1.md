## 0. 一句话定位

**把生物线路当神经网络来「设计 + 预测」的纯前端开放工作台**——面向所有合成生物学团队,让人把基因线路按「输入层(感受元件)→隐藏层(启动子/受体串扰)→输出层(报告元件)」搭出来,正向预测它的输入–输出行为,反向按目标功能训练参数、推荐元件;结构真实、数值示意+注明待标定,可部署到 iGEM 服务器、开源、离线可用。

---

## 1. 愿景与边界

### 1.1 我们要做什么
1. 一个**通用平台**:不绑死任何单一应用场景,任何 synbio 队都能用自己的元件搭网络、做预测与设计。
2. 把团队「第一次讨论」里的四件北极星功能落地:① 元件↔ANN 层级对应 ② 按需求/功能推荐元件 ③ 元件间关系量化 ④ 结果预测。
3. 用团队自己的应用示例进行 dogfooding,验证平台是否真的能服务具体合成生物学设计任务；应用示例可替换,不定义平台边界。

### 1.2 明确不做（非目标,至少 v1 不做）
1. 不做湿实验级的定量预报。平台给的是**概念级、可复现的设计推演**;真要指导湿实验,需用户用自己标定的真值替换示意参数。
2. 不做序列设计 / 密码子优化 / 结构预测(那是 Benchling、Cello、SnapGene 等的领域)。我们做的是「**线路层**的功能建模与设计」。
3. 不做后端 / 账号 / 云存储(iGEM 只静态托管,见 §5)。分享靠可复现的 JSON 文件 + 开源仓库。

### 1.3 服务对象
1. 主用户:iGEM / synbio 团队的建模与软件成员——想在湿实验前先把线路逻辑想清楚、跑通预测。
2. 次用户:评审、教学者、想复用工具的其他团队。

## 3. 产品形态与核心用户旅程

平台是一个单页工作台,围绕五条核心旅程(= 主线功能集):

1. **搭线路**:从元件库拖元件进「输入/隐藏/输出」三层,连边=串扰/调控关系,得到一张可编辑的多层网络。
2. **看预测**(正向):设输入浓度/信号,实时看每层活性、三色荧光/任意输出、时间过程、剂量响应、真值表热图;红色输出可挂双稳态记忆锁。
3. **做设计**(反向):给一张目标真值表(输入组合→期望输出),平台训练拟合出需要的权重/参数,并告诉你「需要多强的启动子、哪些元件」。
4. **推元件**:给定目标功能,从元件库按参数匹配推荐候选元件(感受元件 Km 匹配、启动子强度/正交性匹配等)。
5. **存 / 分享 / 部署**:导出可复现 JSON、导出图、开源、上 iGEM 服务器。

> **MVP「主线」最小可用闭环** = 旅程 1+2(搭 + 正向预测),先把「设计-预测工作台」立起来;旅程 3+4(训练/推荐)紧随其后,是平台的差异化亮点;旅程 5 贯穿始终。

## 4. 概念模型（技术心脏)

### 4.1 生物线路 ↔ 神经网络映射
| 神经网络 | 生物对应 | 平台中的对象 |
|---|---|---|
| 输入层节点 | 感受元件(sensor),把分析物浓度编码成信号(如 AHL) | sensor 节点 |
| 隐藏层节点 | 计算菌内非正交启动子/受体,整合多路信号 | hidden 节点 |
| 权重矩阵 W¹ | 信号↔启动子的**串扰强度**(非对角=非正交) | 输入→隐藏 的边/矩阵 |
| 激活函数 | 启动子的非线性响应(Hill / sigmoid) | 节点 activation |
| 输出层节点 | 报告元件(荧光蛋白等) | reporter 节点 |
| 记忆 | 双稳态锁(越阈即锁) | reporter 的 latch |
| 训练 | 结果导向调启动子强度/受体数/质粒拷贝 | 反向拟合 W、b |

### 4.2 数学规范（草案,后续拆 `02_引擎规范.md`)
1. 感受:`s_k = Hill(c_k, Km_k, n_k)`,`Hill(x,Km,n)=x^n/(Km^n+x^n)`(x<0 截 0)。
2. 逐层前向:`pre_j = Σ_k W[l][j,k]·a_k + b_j`;`drive_j = act(pre_j)`(act∈{linear, hill, logistic})。
3. 动力学(可选,逐节点一阶):`dx_j/dt = α_j·drive_j − β_j·x_j`,稳态 `x_ss = (α/β)·drive`,欧拉积分,节点值 ≥0。
4. 三级读出 / 记忆:对指定输出节点 `Z<θ1 绿 / θ1≤Z<θ2 黄 / Z≥θ2 红`;`latch` 节点越 θ2 即锁。
5. **demo3 特例**:输入层=3 个 sensor(act=hill),无隐藏层,输出=1 个 linear+dynamics+latch 节点,有效权重 [0.96,0.72,1.20],α/β/θ 同 demo3 → 引擎须逐点复现(单测:P=[1,0,0] 稳态 Z_ss=1.92,越 θ2 锁红)。

### 4.3 元件库数据模型（草案,后续拆 `03_元件库schema.md`)
每个元件:`{ id, 名称, 类型(sensor|qs-signal|promoter|rbs|regulator|reporter|logic), 可占据的层, 参数{...} + 每个参数的「来源」与「是否待标定」标记, iGEM Registry 编号/文献链接(若有), 备注 }`。
1. sensor:分析物、Km、Hill n、输出信号物种。
2. promoter:强度、响应信号、**串扰谱**(哪些信号以多大相对强度激活它)。
3. reporter:输出颜色/通道、亮度、成熟时间。
> 关键原则(贯穿全库):**结构真实、数值示意 + 注明待标定**;能查到真值就用 iGEM Registry/文献的真参数,并强制记录来源(见 §8 防幻觉纪律)。

### 4.4 关系/串扰模型
1. 「元件间关系量化」= 权重矩阵的矩阵元(串扰强度、调控强度)。
2. 数据现实:真实的启动子串扰定量数据极少。平台的姿态是**提供「表达 + 仿真关系」的框架**,数值由用户/文献提供或标记为待标定,而不是假装我们有一张权威串扰表。

### 4.5 训练 / 反向设计
1. 输入:目标数据集(输入组合→期望输出),可来自真值表编辑器。
2. 方法:小网络,梯度下降 / 进化搜索(CMA、随机重启)均可,先求能跑通且可解释。
3. 输出:拟合后的权重/参数 + 「翻译」成生物语言的建议(启动子强度档位、受体数量级、推荐元件)。
4. **诚实边界(写进界面)**:基于示意参数的训练是**设计推演**,不是湿实验预报。

## 5. 系统架构

1. **形态**:纯前端生产级 SPA(Vite + React + TypeScript),全部计算在浏览器,**静态构建 → 部署到 iGEM 服务器**;离线可用、零运维、别队 clone 即用。
2. **模块分层**:
   - `engine/`:与界面无关的纯 TS 引擎(前向预测、动力学、训练、分类)。可单测、可被任何 UI/脚本复用。
   - `data/`:元件库(打包 JSON)+ schema 校验 + 来源元数据。
   - `state/`:工程状态(网络、参数、视图),可序列化为可复现 JSON。
   - `ui/`:组件化界面(元件库、网络编辑器、预测视图、训练面板、推荐面板、导出)。
   - `io/`:导入导出(JSON/CSV/PNG)、示例工程、部署产物。
3. **从 demo3 提炼复用**:`kernel.js`→升级为多层引擎(单神经元为特例)、`theme.css` 设计 tokens、`chart.js`、网络原理图视觉语言。
4. **设计系统**:沿用 demo3 的 ASSAY 台式控制台语言(浅/深双主题、信号仪表、示波曲线),组件化重写。

## 8. 数据来源与防幻觉纪律（红线)

> 背景:AI 在生物文献/元件号上极易「一本正经地编造」。本工程对此**零容忍**。

1. **来源强制**:元件库里任何具体元件号、参数值、串扰数据、文献,**必须带可核实来源**(Registry URL / DOI / 论文链接)。无来源的一律标记 `待核实/示意值`,不得呈现为权威。
2. **结构真实、数值示意**:平台默认承认数值是示意/可调的;真值是「能查到才用」,查不到就留接口给用户标定。这一条是产品原则,也是诚实声明。
3. **调研流程**:M2 的元件调研以「**先列候选类别 → 逐个查证 → 录入并附来源 → 标记置信度**」推进;我会把候选与来源整理成清单交你/师姐过目,确认后才进库,绝不直接把 AI 生成的参数当数据。
4. **审计可追溯**:每个数值在库里都能点开看「从哪来、谁核实、是否待标定」。
